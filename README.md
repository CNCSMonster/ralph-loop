# Ralph Loop

极简自动化开发循环 - 只需一个 `PROMPT.md`，让 AI Agent 自动完成开发任务。

## 安装

单脚本，无需依赖，复制到 `~/.local/bin` 即可全局使用：

```bash
cp ralph-loop ~/.local/bin/ralph-loop
chmod +x ~/.local/bin/ralph-loop
```

确保 `~/.local/bin` 在你的 `PATH` 中：

```bash
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
```

安装后任意目录可直接运行 `ralph-loop` 命令。

## 原理

核心就是一个 `while` 循环，每次全新上下文启动 AI，通过文件系统持久化状态。

## 快速开始

### 1. 编写需求

创建 `PROMPT.md`，写入你的项目需求。

### 2. 初始化

```bash
ralph-loop init
```

自动生成 `.ralph/` 下的约束文件和状态文件模板（不会覆盖已有文件）。

### 3. 启动循环

```bash
ralph-loop run        # 默认 50 次迭代
ralph-loop run 100    # 指定 100 次
```

### 4. 配合 AI

脚本会自动启动 `qwen --yolo` 执行任务，无需人工干预。

如果需要在迭代之间暂停观察，可以在 qwen 执行完成后按 `Ctrl+C` 中断脚本。

### 5. 断点续跑

如果迭代次数用完但任务没完成，直接再次 `run` 即可，会从上次进度继续：

```bash
ralph-loop run 20     # 继续之前的进度
```

## 命令

| 命令 | 说明 |
|------|------|
| `ralph-loop init` | 初始化项目结构（幂等，不覆盖已有文件） |
| `ralph-loop run [N]` | 启动循环，N 为最大迭代次数（默认 50） |
| `ralph-loop status` | 生成状态分析 prompt，交给 qwen-code 分析 |
| `ralph-loop help` | 显示帮助 |

## 文件结构

```
.
├── PROMPT.md                     # 总任务描述（唯一需要你写的）
├── ralph-loop                    # 主脚本
└── .ralph/
    ├── constraints.md            # AI 约束规则（自动生成，❌ 禁止修改）
    ├── plan.md                   # 计划与子任务状态（✅ AI 更新）
    ├── progress.md               # 进度记录（✅ AI 更新）
    ├── verify.md                 # 验证记录（✅ AI 更新）
    └── docs/                     # 子任务 spec（✅ AI 可选）
```

## 工作流程

```
PROMPT.md（总任务）
  ↓
ralph-loop run
  ↓
while 循环 N 次:
  1. AI 读取 constraints.md + plan.md + progress.md
  2. AI 判断是否需要调整 plan.md
  3. AI 选择下一个子任务 → 实现 → 子任务验证（仅验证当前子任务）
  4. AI 更新 plan.md（标记 passes:true/false）+ progress.md
  5. 如果所有子任务 passes:true → 进入整体验证
  6. 整体验证通过 → verify.md 验证次数+1
  7. 整体验证失败 → 次数归零，修复问题重新开始
  8. 脚本规则检查:
     - 全部子任务通过 且 整体验证3次 → 结束
     - 其他 → 继续

## PROMPT.md 编写规范

`PROMPT.md` 是你与 AI Agent 沟通的唯一入口，写好它至关重要。

### 核心原则

1. **明确目标**：用一句话说明项目要做什么
2. **描述当前状态**：已有代码/文件结构，哪些已完成哪些待做
3. **列出具体要求**：功能点、技术栈、约束条件
4. **定义验证方式**：如何判断任务完成（测试命令、手动检查项等）
5. **提供上下文**：相关背景信息、参考文档、已知问题

### 推荐结构

```markdown
# 总任务名称

## 总任务描述
一句话描述总任务要做什么。

## 当前状态
- 已有：列出已有文件/功能
- 待完成：列出需要实现的子任务
- 已知问题：列出已知 bug 或限制

## 具体要求
1. 功能需求 1
2. 功能需求 2
3. 技术约束（如：不引入外部依赖、使用 Python 3.10+ 等）

## 验证方式
- 命令 1 → 期望输出/行为
- 命令 2 → 期望输出/行为
- 其他检查项

## 提示
- 任何有助于 AI 理解项目的信息（如相关文档链接、设计决策原因等）
```

### 示例：CLI 工具

```markdown
# 待办事项 CLI

## 总任务描述
创建一个简单的命令行待办事项管理工具。

## 当前状态
- 空项目，只有 README.md
- 需要从零开始实现

## 具体要求
1. 支持添加/完成/删除/列出待办事项
2. 数据持久化到 JSON 文件
3. 使用 Python 编写，不引入第三方依赖
4. 每个命令有清晰的输出格式

## 验证方式
- `todo add "买牛奶"` → 输出 "已添加: 买牛奶"
- `todo list` → 列出所有未完成事项
- `todo done 1` → 标记第 1 项为完成
- `todo list --all` → 列出所有事项（含已完成）

## 提示
- 数据存储格式建议：`{"items": [{"id": 1, "text": "...", "done": false}]}`
- 命令使用 argparse 解析
```

### 示例：Web 应用

```markdown
# 个人博客

## 总任务描述
使用 Flask 构建一个简单的个人博客系统。

## 当前状态
- 已有基础的 app.py 和 templates/ 目录
- 数据库模型已定义（Post 表）
- 缺少首页、文章详情、创建文章功能

## 具体要求
1. 首页显示文章列表（按时间倒序）
2. 点击文章标题进入详情页
3. 支持 Markdown 格式渲染
4. 管理员可创建/编辑文章（简单密码保护）
5. 使用 SQLite 存储数据

## 验证方式
- 访问 `/` → 显示文章列表
- 访问 `/post/1` → 显示文章详情
- 访问 `/admin` 输入密码后 → 显示管理界面
- 创建文章后刷新首页 → 新文章出现在列表

## 提示
- Markdown 渲染可使用 `markdown` 库（已安装在 requirements.txt）
- 密码保护用简单的 session 机制即可，不需要完整认证系统
```

### 避免的写法

❌ **太模糊**："做一个好用的工具"
✅ **明确**："创建一个支持自动补全的命令行文件搜索工具"

❌ **缺少验证标准**："优化性能"
✅ **可测量**："将启动时间从 2s 降低到 500ms 以内"

❌ **假设 AI 知道上下文**："把之前的 bug 修一下"
✅ **提供上下文**："修复 status 命令统计 bug：init 后显示总任务数为 2，实际 plan.md 中没有任务定义"

## 约束机制

`.ralph/constraints.md` 定义 AI 必须遵守的规则：
- 调整计划 → 选择子任务 → 实现 → 子任务验证 → 更新状态 → 提交
- **只有此文件禁止 AI 修改**，其他状态文件 AI 自由读写
- `plan.md` 有格式约定，脚本通过正则 `passes:(true|false)` 统计子任务状态

## 完成条件

循环在以下情况结束：
- ✅ `.ralph/plan.md` 中所有子任务 `passes:true` **且** `.ralph/verify.md` 验证次数达到 3 次（整体验证连续通过）
- ❌ 达到最大迭代次数
- 用户输入 `quit`

## 最佳实践

### 编写高效的 PROMPT.md

1. **保持聚焦**：每次迭代只关注一个明确的目标，避免大而模糊的需求
2. **量化验证标准**：用具体的输出和行为作为完成标准，而不是主观判断
3. **渐进式开发**：将总任务拆分为子任务，让 AI 逐个完成并验证
4. **提供充分上下文**：包括技术栈、约束条件、已知问题、参考文档等

### 合理设置迭代次数

- 小型任务（如修复 bug、添加简单功能）：5-10 次
- 中型任务（如实现一个模块）：10-30 次
- 大型任务（如从零构建应用）：30-50 次
- 不建议设置过高，AI 通常会在完成任务后自动结束

### 善用 status 命令

在迭代之间使用 `ralph-loop status` 可以：
- 查看当前任务进度和统计
- 让 AI 分析项目状态并给出建议
- 帮助判断下一步行动

### 管理计划文件

- `plan.md` 是 AI 的工作计划，你也可以手动编辑调整
- 子任务优先级用 `priority:N` 表示，数值越小越优先
- 验证状态用 `passes:true/false` 标记，脚本自动统计
- 如果发现 AI 偏离方向，手动调整 plan.md 重新引导

### 隔离测试

- 所有验证和测试应在临时目录或 `.test/` 目录进行
- 不要在生产目录直接运行测试脚本
- 项目自带的 `test-ralph-loop.sh` 使用临时目录，安全可信赖

### 断点续跑策略

- 如果 AI 输出不理想，可以调整 PROMPT.md 后重新 run
- 状态文件保留历史记录，帮助 AI 理解上下文
- 必要时可以手动清理 `.ralph/` 目录重新开始

## 常见问题

**Q: 可以中断后继续吗？**
A: 可以。状态持久化在文件中，随时 `ralph-loop run N` 即可续跑。

**Q: 会覆盖我之前写的进度吗？**
A: 不会。`init` 是幂等的，已有文件不会覆盖。

**Q: 如何自定义验证命令？**
A: 在 `PROMPT.md` 中说明项目使用的测试/验证方式，AI 会按照约束规则执行。

**Q: 总任务和子任务有什么区别？**
A: 总任务是你在 `PROMPT.md` 中描述的完整需求目标。AI 会自动将总任务分解为多个子任务，记录在 `plan.md` 中，然后逐个实现和验证。

**Q: 什么是子任务验证和整体验证？**
A: 子任务验证是每完成一个子任务后的局部验证，只标记该子任务是否通过。整体验证是所有子任务完成后，对整个总任务的最终验证，连续通过 3 次才算总任务完成。
